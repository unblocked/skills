# Link Resolver

This skill encodes the operating invariants of `link_resolver`: what inputs it expects, what kinds of URLs it can hydrate, how resolved content is shaped, and how to react to partial or failed resolution.

## Input

```text
urls: one URL per line
verbose: optional; set to "true" for extra detail like PR diffs or Jira comments
```

Examples:

```text
https://github.com/unblocked/unblocked/pull/28157
https://getunblocked.slack.com/archives/C07MRPZB95X/p1770057346161749
```

For multiple URLs, put each one on its own line:

```text
https://github.com/unblocked/unblocked/pull/28157
https://docs.google.com/document/d/abc123/edit
https://getunblocked.slack.com/archives/C07MRPZB95X/p1770057346161749
```

## Core Principles

- Use this tool only when actual supported URLs are present or when a linked artifact must be hydrated.
- Provide exactly one URL per line.
- The exact supported URL patterns are runtime-generated. Do not assume this skill's examples are exhaustive.
- This tool is for integrated URL hydration, not for local file paths or general web browsing.
- Use `verbose` only when extra detail is required.
- Treat failures as structured signals about support, permissions, or partial coverage.

## Supported URL Boundaries

- Typical supported services include Slack, Jira, Confluence, Google Docs, Notion, GitHub, Asana, Coda, Linear, and similar integrated systems.
- The skill cannot hardcode the exact supported regex patterns because they are generated by the tool at runtime.
- If the tool fails to resolve a URL, use the returned "Supported URL Patterns" section as the current source of truth.
- If you need local repository context without a URL, use a different tool.

## Resolution Process

1. Collect the exact URLs that need hydration.
2. Put each URL on its own line in `urls`.
3. Decide whether `verbose` is needed for diffs, comments, or richer artifact detail.
4. Review each resolved section independently.
5. Extract the useful payload: metadata, body content, comments, diffs, embedded images, and follow-on links.
6. If a resolved artifact points to another URL you need, run the tool again with that URL.
7. If only some URLs resolve, continue with the successful results and explicitly call out the failures.

## Using the Output

Resolved output may include hydrated metadata, body text, comments, diffs, images, and other artifact-specific content.

- Treat each resolved URL as its own section of evidence.
- Use the hydrated content to answer precisely instead of paraphrasing from the bare URL.
- Treat resolved artifacts as external or historical context; they may not match the current local workspace or the latest remote state.
- Pull out linked artifacts and concrete identifiers when a second retrieval step is needed.

## Failure Handling

- Unsupported URLs: the tool returns a supported-pattern list. Use that response instead of guessing what should work.
- Slack access failures: if the tool says Unblocked is not a member of the channel, tell the user that and point them to the Slack integration/settings link returned by the tool.
- Partial resolution: use what resolved, list what did not, and continue.
- Other failures: surface the unresolved URLs and say what additional access or follow-up is needed.
